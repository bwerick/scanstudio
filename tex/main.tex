\subsection{Live Recording \& Real-time Feedback}

We use a $4$k Insta360 Link $2$ webcam clamped to a standard reading light to capture overhead video of a person flipping through pages (Figure~\ref{fig:page1}). Video is recorded at $24$ frames per second with automatic focus, ensuring sharpness across frames. The goal of this setup is high replicability: nothing about the hardware or lighting should require specialized equipment. To that end, we support a variety of cameras, including All-in-One cameras like GoPros, smartphones with stands, and other high resolution handheld devices.

The following elements are critical for consistent capture:
\begin{itemize}
    \item \textbf{Lighting:} Clear, even illumination from a desk lamp or overhead light, avoiding glare or shadows.
    \item \textbf{Frame quality:} Video should be recorded at a minimum of 4K resolution, so small characters are legible. 
    \item \textbf{Hands-free operation:} The mount or clamp must be steady, so the user can flip pages naturally without having to hold or operate anything else.
\end{itemize}


Each of the requirements above can be relaxed, but doing so incurs a corresponding need for technical innovation. For example, lighting conditions could be less controlled if we integrate adaptive histogram equalization~\cite{pizer1987adaptive} or low-light denoising models~\cite{li2015low}. Camera movement could be tolerated with the introduction of real-time stabilization~\cite{castillo2004real} or motion-compensated frame selection~\cite{huang2009correlation}. Hands-free operation could be achieved with wearables and AR headsets~\cite{mann1997wearable, Meta2023RayBanMeta}. These are promising directions for future work.

To cite past reviewers of Yifan Wu, is there evidence of self-serializtion? Does feedback from the system cause the user to adjust their behavior, so that system performance improves without system or software level modifications? There's some evidence of that with the capture confirmation and bells, can improve page flipping rate, page crispness by allowing for stillness.

\subsection{Motion Signals \& Keyframing}

Frame selection:
With video: if we are using a video as our input, we want to collect exactly one frame for each set of unique pages from the document we are hoping to extract. When recording the video of a document, there is a predictable pattern that we can exploit to search for these frames. We noticed that from start to finish, there are periodic segments of high motion as we turn the pages. These segments of high motion give us boundaries that tell us where the frames we want are. In the case of a book, we start recording with the cover in frame. We then open the front cover to reveal the back of the covers and the first physical page. When the hands enter the frame and grab the cover and open the book, these motion cause  noticeable differences between pixel values from one frame to the next. As the page is turned, the hand and the edge of the page also get physically closer to the camera which affects the focus, and shadows darken areas of the book and table. When the page is flat, the differences between frames are lesser. We calculate the mean absolute pixel difference for each frame, and when plotted, we can use this as a type of signal. Here is the motion signal of a book with 262 unique frames. In the second plot, each shaded region likely contains a single frame we want that has the desired pages in sharp focus.

\subsubsection{Phase 1: Motion Signal}
The frame to frame mean abosulte pixel differece is computed at reduced resolution to produce a an activity signal that captures page turns as peaks and stable spreads a valleys.
Input: .mp4 .mov video file
Output:
    motion signal numpy file - raw frame motion scores
    smoothed signal.npy - 15 frame rolling average
    metadata.json - video info, resolution, fps, frame count
    motion plot.png - visual

\subsubsection{Phase 2: Detect Peaks}
Page turn events are identified using \texttt{scipy.signal.find\_peaks} on teh smoothed motuion signal. These peaks serve as boundaries where we can select exactly one frame to represent a a unique page/book spread.
Input: smoothed signal and metadata
Output:
    peaks.npy - frame indices of page turns
    spreads.json - list of boundaries, start, end, duration
    peaks metadata.json - parameters used, results info
    peaks plot.png - motion signal w/peaks marked


\subsubsection{Phase 3: Select Keyframes}
A signle best frame is selected for each spread. The best frame will have the lowest motion value which indicates maximum stillness. If multiple frames have an equal motion score, then image sharpness is used as a tie breaker. 

Input: spreads.json, motion folder, video file
Output:
    spreads - keyframes folder holding all selected frames, each full resolution
    keyframes.json - metadata for each frame (idx, motion, step)
    selection\_plot.png view where in plot frames are

\subsubsection{Phase 5: Crop Keyframes}
Removes table and background from teh keyframe images. It supports videos with two spages in frame or single pages in frame
Otsu page detection- convert frame to grayscale, seperate bright page from dark background, find the contours of all white reagions, find largest white region which is probably the page, fit rectangle around that largest region and crop it out with given margin
\begin{itemize}
    \item \textbf{Input:} \texttt{keyframes.json}, keyframe images, \texttt{review\_log.json} (for crop bounds)
    \item \textbf{Output:}
    \begin{itemize}
        \item Keyframe images cropped in-place in \texttt{images/}
    \end{itemize}
\end{itemize}


\subsubsection{Phase 6: Split Pages}
Frames where two pages were places side by side are split into individual pages
Input: final-keyframes.json, review-log.json + final-keyframes folder containing images
Output: pages folder, keyframes are split into left and right images
pages.json - ordered list with metadata

\subsubsection{Phase 7: Page Quality Review (Optional)}
Interactive GUI for inspecting final page images and flagging quality issues (crop problems, poor image quality). Produces a shareable report for algorithm improvement.
Input: pages.json + page images
Output: page-review.json - annotations about the quality of the cropping
page-review-report.md - shareable report

\subsubsection{Phase 8: Binarize (Optional)}
Converts page images to black and white using adaptive Gaussian thresholding, computing a per-pixel threshold based on local neighborhood intensity. Suitable for text-heavy documents on thick paper; not recommended for pages with photographs or colored backgrounds.
Input: pages.json + pages images
Output: new folder containing all images from pages.json but in black and white
bw-metadata.json

\subsubsection{Phase 9: Build PDF}
Assembles individual page images into a PDF document using ReportLab, sizing each page to match the source image aspect ratio.
Input: pages.json + page images or bw images
Output: book.pdf - complete pdf
